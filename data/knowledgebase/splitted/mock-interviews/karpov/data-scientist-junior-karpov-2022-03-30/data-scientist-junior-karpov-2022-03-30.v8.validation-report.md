## Как считаются метрики

**Tolerance** — порог **±120 с** для сравнения тайм-кодов: `Δt = время interviewer_question` − маркер главы YouTube`. Если |Δt| ≤ tolerance, item **засчитывается** в покрытие этой главы. Значение задаётся флагом `--tolerance` (в `splitter_post.sh` по умолчанию **120**).


**Покрытие (coverage)** — доля **вопросных** глав YouTube (заголовки в `video.md`, без подводок 🔗), для которых найден **хотя бы один** item с `interviewer_question.time` в пределах **±120 с** от маркера главы. Вступления, разборы, конец и подводки в знаменатель **не входят**. Сводка по главам — **в таблице ниже**; полные тексты всех items — в разделе «Все извлечённые Q&A».


**Согласованность тем (topic consistency)** — среди items, попавших в покрытие (сопоставлены с вопросной главой в ±tolerance), какая доля имеет `question_topic` из списка, разрешённого для **секции** YouTube (`Секция «…»` в Description `video.md`).

Карта секция → допустимые `question_topic`:
- **Python** → `Python`
- **A/B-тесты** → `Experimentation`, `Statistics`
- **Работа с данными** → `SQL`, `Python`
- **ML алгоритмы** → `ML`

**Окна глав** (раздел «Q&A по интервалам глав») — другая ось: item попадает в главу, если `interviewer_question.time` ∈ [маркер главы … маркер следующей главы). Уточняющие вопросы внутри секции (например item #8 между #7 и #9) **здесь видны**, даже если они **не** ближайшие к маркеру в ±tolerance.


**Шаг 5 (LLM-валидация)** — после детерминированного отчёта модель читает приложение в конце файла и для каждой **вопросной** главы оценивает: правдоподобны ли тайм-коды и нет ли явного расхождения смысла с заголовком. Это **не** дублирует coverage; фокус на качестве полей JSON, а не на «попал ли вопрос в соседнюю главу на ±1 с». Результат подмешивается в детальные секции при `--llm-json`.

## Как работает валидация

Валидатор сравнивает разбивку Q&A (JSON) с тайм-кодами из описания видео на YouTube. **Важно:** сплиттер при своей работе `video.md` **не видит** — он работает только с транскриптом. Описание видео используется исключительно для независимой проверки результата и является внешним эталоном: тайм-коды расставлены авторами видео вручную.

**Шаг 1 — Разбор тайм-кодов видео.**
Из блока `## Chapters` в описании видео извлекаются тайм-коды с заголовками. Вступления, паузы, разборы задачи и прочие переходы (заголовки типа «Интро», «Перерыв», «A case study» и т.п.) исключаются из проверки — остаются только реальные вопросы.

**Шаг 2 — Окна глав и items.**
Каждая глава YouTube задаёт интервал до следующей главы. Все items, чьё `interviewer_question.time` попадает в интервал, относятся к этой главе. 0 items → не распознан; 1 → один Q&A; 2+ → несколько Q&A.

**Шаг 2b — LLM-валидация (опционально).**
Шаг 5: вход для модели — раздел **«Приложение: LLM-валидация»** в конце **этого же** `.md`. Ответ сохраняется в `*.validation.llm.vN.json` и подмешивается в таблицу при повторном запуске валидатора с `--llm-json`.

**Шаг 3 — Coverage.**
```
Coverage = вопросные главы с ≥1 item в ±120 с / всего вопросных глав (без подводок) × 100
```
**Δt** в сводной таблице — только у первого item группы. Подводки (🔗) не считаются отдельным вопросом. Непокрытые фрагменты транскрипта — отдельный раздел ниже сводной таблицы.

**Шаг 4 — Topic Consistency.**
```
Topic Consistency = items с верной темой / сопоставленные items с известной секцией × 100
```
Проверяет, что поле `question_topic` в JSON соответствует ожидаемой теме секции:

- Секция **Python** → допустимые темы: ['Python']
- Секция **A/B-тесты** → допустимые темы: ['Experimentation', 'Statistics']
- Секция **Работа с данными** → допустимые темы: ['SQL', 'Python']
- Секция **ML алгоритмы** → допустимые темы: ['ML']

**Δt тайм-кодов** (справочно): средний 11.8с, максимальный 67с — в пределах допуска ±120с.
