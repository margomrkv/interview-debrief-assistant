## Назначение

Второй шаг тренировочного пайплайна (после `build-train-hard-skills`): берёт собранный
`train/hard_skills.json` и проставляет золотые лейблы `reference_score` каждому hard-item по
промпту-оценщику ниже. Результат — тот же файл с per-item `reference_score` (6 полей) и
верхнеуровневым блоком `labeling`.

## Промпт-оценщик (criteria_source)

> Ты — опытный интервьюер компании уровня Avito/InDrive, ответственный за оценку компетенций
> кандидатов. Твоя оценка напрямую влияет на решение о найме, поэтому точность и объективность критичны.
>
> **Уровень кандидата задаётся полем `grade` (`junior` | `middle`) — бери его из данных, НЕ
> предполагай.** Калибруй ожидания под этот уровень: junior-ответ оценивается по junior-стандартам,
> middle — по middle. Все якоря шкалы ниже трактуются относительно целевого `grade`.
>
> Твоя задача: оценить ответ кандидата по трём ключевым параметрам — **Factual Correctness**, **Focus**
> и **Clarity**, используя 10-балльную шкалу, откалиброванную на стандарты заданного `grade`.
>
> **Контекст оценки:**
> - Вопрос интервьюера отражает реальные задачи на должности (планирование, приоритизация, работа со
>   стейкхолдерами, анализ данных и т.д.)
> - Оценивая фактическую корректность: проверяй не только отсутствие явных ошибок, но и полноту/точность
>   используемых терминов и подходов
> - Оценивая фокус: определяй, насколько полно кандидат ответил именно на заданный вопрос, и адресовал ли
>   он все части вопроса
> - Оценивая ясность: смотри на структурированность, логичность и понятность изложения; разговорные
>   формулировки при наличии смысла — допустимы, но должны быть достаточно структурированы
>
> **Шкала калибровки (1–10), относительно целевого `grade`:**
> - **9–10 баллов** — Эталонный для уровня ответ. Полнота, точность, исключительная ясность. 10 — образец
>   для других кандидатов; 9 — крайне сильный ответ с пренебрежимо малыми недочётами.
> - **7–8 баллов** — Сильный ответ. Суть передана корректно, логика прослеживается, факты верны с минорными
>   пробелами. 8 — уверенно выше среднего для уровня; 7 — solid для уровня с одной-двумя заметными
>   недоработками (упущенный аспект, неполная аргументация).
> - **5–6 баллов** — Приемлемый, но средний ответ. Тема адресована, но есть пробелы в деталях, структуре или
>   точности; частично может уходить от фокуса. 6 — ближе к уровню; 5 — пограничный.
> - **3–4 баллов** — Слабый ответ. Существенные пробелы в содержании, структуре или фокусе. Заметные
>   фактические неточности или значительная часть вопроса не покрыта. 4 — есть зачатки правильного
>   направления; 3 — преимущественно мимо.
> - **1–2 баллов** — Несостоятельный ответ. Вопрос фактически не адресован, грубые ошибки, отсутствие
>   структуры или полная нерелевантность. 2 — попытка предпринята, но провальная; 1 — ответ непригоден
>   к оценке по существу.
>
> **Anchor-точки для калибровки:**
> - Дробные оценки не используются — только целые числа от 1 до 10
> - 7 — это «нанимаем», 5–6 — «спорно, нужен второй сигнал», ≤4 — «не нанимаем по этому ответу»
> - Не завышай: если сомневаешься между двумя соседними баллами, выбирай нижний
>
> Проанализируй контекст (reference answer если есть, interviewer feedback), а затем выдай:
> 1. **Factual Correctness** (1–10) — оценка точности и полноты информации
> 2. **Focus** (1–10) — оценка того, насколько кандидат ответил именно на вопрос
> 3. **Clarity** (1–10) — оценка структурированности и понятности ответа
>
> Приоритет у тебя именно reference answer если есть, interviewer feedback, а не твои внутренние знания
> модели.

## Политика лейблинга

- **Опирайся только** на `interviewer_feedback` и/или `reference_answer` — не на индустриальный baseline
  и не на внутренние знания модели.
- **`grade` берётся из item**, не предполагается.
- Item, у которого пусты И `interviewer_feedback`, И `reference_answer` → **не оценивать** (попадёт в
  `reference_score: null`, `unscored_reason: "no_interviewer_signal"`).
- Пустой `candidate_answer` → ответ считается несостоявшимся (writer проставит `(1,1,1)` / `aggregate=missing`).

## Процедура

1. Прочитать `train/hard_skills.json` (порядок items фиксирован build-скриптом — индекс 0..N-1 стабилен).
2. Для каждого item применить промпт-оценщик выше с учётом его `grade`. Источник сигнала зафиксировать:
   `signal_source` ∈ {`feedback`, `reference`, `both`}. Краткое (1–2 фразы) обоснование — `rationale`.
3. Записать оценки в `labels.json` (рядом с этим файлом) — список объектов:
   ```json
   [ { "idx": 0, "factual_correctness": 8, "focus": 7, "clarity": 7,
       "signal_source": "feedback", "rationale": "..." }, ... ]
   ```
   Items без сигнала (см. политику) в `labels.json` НЕ включать — writer сам проставит им `null`.
4. Запустить writer:
   ```bash
   python3 .claude/skills/label-train-hard-skills/apply_labels.py
   ```
   Он добавит `aggregate` (из 1–10 якорей), блок `labeling` и перепишет файл. Перезапуск идемпотентен.
5. Sanity-check: writer печатает `labeled=…, unscored=…, total=…`. Убедиться, что у каждого
   `reference_score != null` ровно 6 полей, числа в 1–10, `aggregate` ∈ {strong, adequate, weak, missing}.

## Выходная структура

```json
{
  "split_strategy": "...",
  "labeling": { "labeled_at": "...", "labeler": "...", "scale": "1..10",
                "criteria_source": "...", "policy": "...",
                "stats": { "total": N, "labeled": …, "unscored": … } },
  "items": [ { ..., "reference_score": {
      "factual_correctness": 1..10, "focus": 1..10, "clarity": 1..10,
      "aggregate": "strong|adequate|weak|missing",
      "signal_source": "feedback|reference|both", "rationale": "..." } } ]
}
```
